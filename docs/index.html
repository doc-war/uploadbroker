定位

临时资源管理的轻量级代理网关，做现有存储（Local / S3 / OSS / R2）的统一安全外壳。

生命周期

专为 TTL 1~24h 设计，后台秒/分级主动清理。契合 AI 推理（图生图、语音转文字）高频短命临时文件的刚需。

安全性

BLAKE2b 内容寻址 + URL 签名 + Salt 轮换，零信任架构。Platform 不需要知道底层存储结构，URL 不暴露真实桶路径。

体积与依赖

Go 单二进制 + SQLite，极简内聚。极易集成进 CI/CD 或作为 Sidecar 部署。

FAQ

为什么不直接用主流框架？

UploadBroker 不做底层存储，而是在上层提供统⼀的安全代理层。以下对比帮你理解它跟主流存储方案的本质区别。

MinIO 已经是 S3 兼容的对象存储了，为什么还要套一层 Broker？

底层存储 MinIO 定位是完整的对象存储系统，而 UploadBroker 定位是资源代理网关，它们解决的是不同层次的问题。

  • MinIO：持久化存储海量数据，面向长期存储场景（年级生命周期），需要维护集群、配置桶策略和 Presigned URL 权限
  • UploadBroker：管理 AI 推理产生的临时文件（1~24h TTL），自动清理过期资源，统一安全层（魔数检测 + URL 签名 + 扩展名校验），存储后端对业务透明

MinIO 的 Presigned URL 暴露了桶路径和权限模型，而 Broker 的 URL 零信任——Platform 只需拿到标准 URL，不需要知道底层是 MinIO、OSS 还是本地磁盘。切换存储后端时上层零改动。

一句话总结：MinIO 是做存储的，UploadBroker 是做安全的——它始终站在存储的前面，充当统一的安全代理。

SeaweedFS 支持 FUSE、S3、NoSQL，为什么不直接用？

底层存储 SeaweedFS 是一个功能丰富的分布式文件存储，适合大规模持久化数据场景。

  • SeaweedFS：Master + Volume 架构，运维成本高，无原生短 TTL 机制，过期清理需外部脚本来做
  • UploadBroker：单二进制，0 运维，内建秒级 TTL 清理，底层存储只负责读写，生命周期由 Broker 统一管理

SeaweedFS 的 FUSE 挂载功能对 AI 训练场景有价值，但对于 API 驱动的临时文件读写（推理服务的输出图片、语音、PDF），Broker 更轻量、更聚焦。

Broker 甚至可以以 SeaweedFS 为底层存储（通过 S3 兼容接口），在其上叠加安全、TTL 和路由能力。

一句话总结：SeaweedFS 是通用分布式文件系统，UploadBroker 是专为 API 临时文件场景设计的轻量资源代理。

Ceph 是业界标准分布式存储，为什么不直接用？

基础设施 Ceph 是重量级的企业级分布式存储，提供块、文件、对象三种接口。

  • Ceph：部署复杂（MON/MGR/OSD/MDS），运维门槛极高，适合大规模持久化存储（PB 级）
  • UploadBroker：单二进制，开箱即用，适合 AI 推理场景中高频产生、短生命周期的临时文件

Ceph 在超大规模集群中表现出色，但 AI 推理场景的临时文件管理需求是：上传 → 消费 → 过期清理，这个循环只需要小时级存在。用 Ceph 管这种临时数据是"杀鸡用牛刀"。

UploadBroker 让你以单二进制部署即可获得企业级安全防护（魔数检测、URL 签名、TTL 管理），不需要为临时文件搭建重量级分布式存储系统。

一句话总结：Ceph 是为大规模持久化设计的，UploadBroker 是为临时文件场景设计的——需求层级不同。

Garage 轻量且 S3 兼容，为什么不用？

轻量存储 Garage 是一个优秀的轻量级 S3 兼容对象存储，部署简单，适合自托管场景。

  • Garage：轻量的底层存储节点，专注于数据持久化和复制，但缺少资源生命周期管理和安全防护层
  • UploadBroker：在存储之上提供 完整的安全链——魔数检测、扩展名校验、URL 签名、HMAC 鉴权、Salt 轮换

如果把 Garage 和 UploadBroker 放在一起看：Garage 是做后端存储的，Broker 是做前端安全代理的。它们完美互补——你可以用 Garage 当底层，Broker 当接入层。

UploadBroker 的独特价值在于：存储后端的可替换性——今天用 Garage，明天切 MinIO，后天切阿里云 OSS，Platform 端的 API 和 URL 格式零变化。

一句话总结：Garage 和 UploadBroker 不是替代关系，而是互补关系——Garage 管存储，Broker 管安全与生命周期。

RustFS 高性能高安全，为什么不直接用？

新兴存储 RustFS 以高性能和内存安全著称，适合 IO 密集型场景。

  • RustFS：专注于底层存储性能和一致性，适合高频读写场景，但缺少针对 AI 临时文件场景的业务抽象
  • UploadBroker：专注 AI 推理工作流——图生图输出 PNG、语音转文字输出 WAV/MP3、视频处理输出 MP4，每个文件天然小时级 TTL

RustFS 的存储性能优势在底层，但 UploadBroker 的价值在上层业务逻辑：内容寻址、MIME 安全洗牌、多存储路由、自动过期清理。Broker 甚至可以用 RustFS 作为底层后端。

性能并非 AI 临时文件管理的瓶颈——通常单机就能承载推理节点的输出流量。真正的痛点是：安全防护、过期管理和存储切换的灵活性，这正是 UploadBroker 的核心能力。

一句话总结：RustFS 优化的是存储性能，UploadBroker 优化的是业务安全与灵活性——不同层次的差异化价值。

2

编写配置 uploadBroker.yaml

3

启动 Broker

4

5

读取文件

# uploadBroker.yaml
listen: 127.0.0.1:9001
base_url: https://upload.example.com
url_blake2b_salts:
  - current-salt
url_prefix: tmp

storage:
  upload_driver: local
  drivers:
    local:
      root: ./objects

# optional hmac_secret for production
# hmac_secret: "platform_broker_shared_token"
$ ./uploadBroker
$ curl -F "file=@photo.png" "http://127.0.0.1:9001/v1/upload"
{"code":0,"data":{"url":"https://...","mimeType":"image/png"}}
$ curl "https://upload.example.com/tmp/..." # 读取文件
